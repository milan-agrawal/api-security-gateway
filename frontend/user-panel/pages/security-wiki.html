Threat Library

Common API attack types, what they look like, and how the gateway responds

7 threats
Critical

Brute Force Attack

An attacker submits massive volumes of requests — often login or verification attempts — systematically trying different values to guess valid credentials, tokens, or access codes.

Gateway response: Blocked
Critical

Credential Stuffing

Attackers use large lists of breached username-and-password pairs from previous data leaks, replaying them against your API to gain unauthorized access to real accounts.

Gateway response: Blocked
High

Request Flooding

A client or botnet sends abnormally high request volumes in a short window — intending to exhaust backend resources and degrade or take the API offline.

Gateway response: Throttled Blocked
High

Endpoint Scanning

Automated tools systematically request many different URL paths to map out undocumented or hidden endpoints that could be targeted in follow-up attacks.

Gateway response: Throttled
Medium

Replay Attack

A previously captured, valid API request is re-sent by an attacker to repeat an action — such as re-triggering a transaction, authentication step, or data modification.

Gateway response: Blocked
Critical

API Key Compromise

A valid API key is leaked — committed to version control, embedded in client-side code, or intercepted — and then used by unauthorized parties to make requests on your behalf.

Gateway response: Blocked
High

Automated Scraping

Bots programmatically extract large volumes of data from your API at a rate far exceeding normal usage — consuming resources and violating fair-use policies.

Gateway response: Throttled

Concepts Glossary

Plain-English definitions of technical terms you'll encounter in logs and documentation

8 terms

Rate limiting enforces a ceiling on how many requests a given API key can make within a defined time period. Once exceeded, further requests receive a 429 Too Many Requests response until the window resets. This prevents abuse, ensures fair access for all consumers, and protects the backend from resource exhaustion.

Example: 10 requests allowed per 60-second rolling window per API key

Anomaly detection identifies requests or traffic sessions that deviate significantly from the established baseline of normal activity. Unlike rule-based checks that only catch known patterns, anomaly detection can surface novel attack methods by recognizing that something is different — even if it isn't explicitly registered as harmful.

Example: A key making 10 req/min suddenly jumping to 500 req/min

Behavioral analysis examines a client's historical request data to build a profile of their typical usage — which endpoints they call, at what rate, how varied their requests are, and how many are blocked. Clients whose current behavior deviates significantly from their own profile are flagged as potentially compromised or abusive.

Signals: endpoint diversity, request rate, error ratio, inter-arrival timing variance

Rather than counting requests in a fixed time bucket (e.g., "during this exact minute"), a sliding window tracks requests in a rolling period ending right now. This closes the "boundary burst" loophole where a client could exhaust their limit at the end of one window, then immediately exhaust the next without any pause.

Example: "Requests in the last 60 seconds" — recalculated on every new request

Throttling (429) means "slow down" — the client exceeded their rate quota. No malicious intent is assumed; they can retry once the rolling window resets. Blocking (403) means "denied" — the gateway determined the request is abusive or malicious based on behavioral signals and will not forward it to the backend.

Throttled → HTTP 429, retry after window resets  ·  Blocked → HTTP 403

When the gateway issues a security verdict it assigns a confidence level indicating how certain it is. High means multiple independent analysis layers agreed on the outcome. Medium means partial agreement. Low means only a weak signal triggered the decision — the verdict may be precautionary and worth reviewing via an appeal if it affected legitimate traffic.

High confidence BLOCK: strong corroborating evidence from multiple analysis layers

A security event is created any time the gateway makes a non-trivial decision on a request — blocking, throttling, or flagging it. Each record includes the request ID, timestamp, IP address, API key, endpoint, the verdict, and the reason. These events are what you see on the Security Logs page in your dashboard.

View on: Security Logs page in your dashboard

Request fingerprinting builds a multi-dimensional identifier for a client by combining several attributes from their requests. This enables more robust identity tracking than IP alone (which can rotate or be shared), helping correlate requests across a session and detect suspicious changes in behavior or origin.

Attributes: IP address, API key, endpoint usage patterns, request rate, timing

Best Practices

Actionable steps to keep your APIs and account as secure as possible

4 guides
API Key Security

Protect your API keys

  • Never include API keys in frontend code, URLs, or public repositories
  • Use a separate key per service or environment — never share a single key everywhere
  • Rotate keys periodically and immediately after any suspected exposure
  • Revoke unused or old keys — fewer active keys means a smaller attack surface
Account Protection

Secure your account access

  • Enable Two-Factor Authentication — the single most impactful security step
  • Store your 2FA backup recovery codes somewhere secure and offline
  • Review your active sessions regularly and revoke any you don't recognise
  • Use a strong, unique password and change it immediately if you suspect a breach
Monitoring

Stay on top of security events

  • Check your Security Logs regularly — even brief scans can surface issues early
  • Clusters of BLOCKED events from the same IP signal a targeted attack
  • Sudden spikes in your own key's usage may indicate it has been compromised
  • Use request IDs to trace specific events and correlate them across log entries
When Blocked

What to do when a request is rejected

  • A 429 means rate limited — check your Usage Plans page and wait for the window to reset
  • A 403 means the gateway flagged the request — find the reason in your Security Logs
  • If the block was incorrect, use the request ID from your logs to submit an appeal
  • Persistent 403s on legitimate traffic may mean an unusual burst pattern — spread requests more evenly

Understanding Your Logs

A quick-reference guide to every verdict and field in the Security Logs page

Every entry in your Security Logs represents a single request the gateway processed. Use this reference to understand what each decision value means and what every field contains — so you can quickly identify issues, trace specific requests, and spot abnormal patterns.

Verdict Reference
Decision Status Meaning Triggered by
ALLOWED 200 Passed all checks — forwarded to the backend API normally. Clean, legitimate traffic within quota and within normal behavioral bounds.
BLOCKED 403 Rejected at the gateway — never reached your backend. Detected abuse pattern, rule violation, or high-confidence anomalous behavior.
THROTTLED 429 Rate limit exceeded — window resets automatically, no action needed. Request count for the rolling window exceeded the quota for this API key.
FLAGGED 200 Allowed but marked as suspicious for review. Not severe enough to block. Mild anomaly signal detected without enough corroboration to warrant blocking.
Status codes: 200 OK · 403 Forbidden · 429 Too Many Requests
Field Reference
Field Description Example
request_id Unique identifier assigned to every request. Use this when referencing an event or submitting an appeal. a3f8…d21c
timestamp Date and time the request arrived at the gateway, in UTC. 2026-03-13 14:22:05

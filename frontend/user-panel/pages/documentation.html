Getting Started

What the gateway does & how to begin

What Is the API Security Gateway?

The API Security Gateway is an intelligent security layer that protects your APIs from threats. Every incoming request is automatically inspected, scored for risk, and either allowed or blocked — all before it reaches your application.

Key Capabilities

  • Threat Detection — Automatically identifies and blocks SQL injection, XSS, path traversal, and other common attacks.
  • Rate Limiting — Prevents abuse by enforcing per-key request limits.
  • ML-Powered Analysis — Uses machine learning to detect anomalous behavior that static rules might miss.
  • Real-Time Monitoring — Every request is logged and analyzed for security insights.
  • Multi-Factor Authentication — Secure your account with TOTP-based two-factor authentication.

Quick Start Checklist

  1. Log in to your account and enable 2FA from your Profile page.
  2. Navigate to API Keys and generate your first key.
  3. Use the gateway base URL provided in your dashboard to send requests.
  4. Include your API key in the X-API-KEY header with every request.
All API traffic must go through the gateway. Direct access to protected resources is not permitted.

Authentication

Sign in, access tokens & multi-factor authentication

Signing In

Authenticate with your email and password to receive an access token. This token is required for all authenticated operations.

Request 
POST /auth/login
Content-Type: application/json

{
  "email": "you@example.com",
  "password": "your-password"
}

On success, you'll receive an access token along with basic account info:

Response 
{
  "token": "eyJhbGciOiJIUzI1NiIs...",
  "role": "user",
  "full_name": "John Doe",
  "email": "you@example.com"
}

Using Your Access Token

Include the token in the Authorization header for every authenticated request:

Header 
Authorization: Bearer <your-access-token>

Two-Factor Authentication (2FA)

If 2FA is enabled on your account, the login endpoint returns a temporary token and an mfa_required flag. You'll need to verify with your authenticator app before receiving the full access token.

Response (2FA Required) 
{
  "mfa_required": true,
  "temp_token": "eyJhbGciOiJIUzI1NiIs..."
}

Complete the verification by submitting the 6-digit code from your authenticator:

Request 
POST /auth/mfa/verify
Authorization: Bearer <temp_token>
Content-Type: application/json

{
  "code": "123456"
}
We strongly recommend enabling 2FA from your Profile page. It adds a critical layer of protection to your account.

API Keys

Managing keys, usage guidelines & security best practices

What Is an API Key?

An API key is a unique credential that identifies and authenticates your application when sending requests through the gateway. Every request must include a valid API key.

Including Your API Key

Pass your key in the X-API-KEY header:

Header 
X-API-KEY: your-api-key-here

Managing Keys

You can create, view, and revoke API keys from the API Keys page in your dashboard. Each key can be individually activated or deactivated.

Security Best Practices

  • Keep keys secret — never commit them to source code, share them publicly, or embed them in client-side applications.
  • Use environment variables — store keys in .env files or a secrets manager.
  • Rotate regularly — periodically generate new keys and retire old ones.
  • Separate environments — use different keys for development, staging, and production.
  • Revoke immediately if a key is compromised — deactivate it from your dashboard without delay.
Requests without a valid API key are immediately rejected with a 403 Forbidden response.

Making Your First Request

Step-by-step walkthrough with code examples

Step 1 — Create an API Key

Go to the API Keys page in your dashboard and create a new key. Copy and store it securely — you'll need it for every request.

Step 2 — Send a Request

Use your gateway base URL along with your API key to send a request. Here are examples in different languages:

cURL 
curl -X GET https://gateway.yourapi.com/api/data \
  -H "X-API-KEY: your-api-key-here"
JavaScript 
const response = await fetch('https://gateway.yourapi.com/api/data', {
    method: 'GET',
    headers: {
        'X-API-KEY': 'your-api-key-here'
    }
});

const data = await response.json();
console.log(data);
Python 
import requests

response = requests.get(
    'https://gateway.yourapi.com/api/data',
    headers={'X-API-KEY': 'your-api-key-here'}
)

print(response.json())

Step 3 — Verify the Response

A successful request returns your data along with a unique request identifier:

Response (200 OK) 
{
  "message": "Protected data retrieved successfully",
  "request_id": "a1b2c3d4-e5f6-7890-abcd-ef1234567890"
}
Each response includes a unique request ID. Save this if you need to reference a specific request when contacting support.

Security & Protection

How the gateway keeps your APIs safe

The gateway provides multiple layers of protection for every request. Security is active by default — no configuration required.

What's Protected

Threat Type Protection
SQL Injection Malicious SQL patterns in query strings, headers, and request bodies are detected and blocked.
Cross-Site Scripting (XSS) Script injection attempts in parameters and payloads are identified and rejected.
Path Traversal Attempts to access unauthorized file paths (e.g., ../../etc/passwd) are blocked.
Brute Force Excessive failed attempts from the same source are rate-limited and flagged.
Anomalous Behavior Unusual traffic patterns are flagged by the ML-based anomaly detection system.
API Abuse Automated scraping, credential stuffing, and excessive requests are detected and throttled.

What Happens When a Request Is Blocked

If a request is determined to be malicious, it's immediately rejected with a 403 Forbidden response. The event is logged and visible in your Security Logs.

Blocked Response 
{
  "detail": "Request blocked by security policy"
}
If you believe a legitimate request was incorrectly blocked, you can submit an Appeal from the Appeal page in your dashboard.

Rate Limiting

Request limits, throttling & how to handle 429 responses

How It Works

To maintain service stability and prevent abuse, the gateway enforces request limits on a per-key basis. Each API key has a maximum number of allowed requests within a rolling time window.

Your current rate limits are visible on the Usage Plans page. Limits may vary based on your plan tier.

When You Exceed the Limit

If you exceed your allowed request count, the gateway returns a 429 Too Many Requests response:

Response (429) 
{
  "detail": "Rate limit exceeded. Try again later."
}

Handling Rate Limits Gracefully

  • Implement exponential backoff — on a 429 response, wait progressively longer before retrying (1s, 2s, 4s, 8s...).
  • Cache where possible — avoid unnecessary API calls by caching data that doesn't change frequently.
  • Batch operations — combine multiple operations into fewer requests when your use case allows it.
  • Monitor usage — keep track of your request count to stay within limits proactively.
Repeatedly exceeding rate limits may cause the anomaly detection system to temporarily suspend your API key.

Error Handling

Status codes, error format & troubleshooting guide

Error Response Format

All errors follow a consistent JSON structure with a human-readable message:

Error Response 
{
  "detail": "Description of what went wrong"
}

Status Code Reference

Status Meaning What To Do
200 OK Request succeeded — process the response normally.
400 Bad Request Check your request body for missing or invalid fields.
401 Unauthorized Your access token is missing, expired, or revoked. Sign in again.
403 Forbidden Your API key is invalid, deactivated, or the request was blocked by a security rule.
404 Not Found The requested endpoint or resource does not exist. Verify the URL.
429 Too Many Requests You've exceeded your rate limit. Wait and retry with exponential backoff.
500 Server Error An unexpected error occurred. If it persists, contact support.
503 Service Unavailable The service is temporarily unavailable. Check the System Status page.

Troubleshooting

  • Getting 401 after changing your password? — Your previous token was invalidated. Sign in again to get a fresh token.
  • 403 even though your key is valid? — The request may have been flagged by the security engine. Review your Security Logs for details.
  • Frequent 429 responses? — Review your request patterns and implement caching or backoff strategies.
  • 503 errors? — The service may be undergoing maintenance. Check the System Status page for updates.
Always read the detail field in error responses — it tells you exactly what went wrong and often how to fix it.
